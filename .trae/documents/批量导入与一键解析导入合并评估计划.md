# 批量导入题目与一键解析并导入合并评估计划

## 结论先行
- **建议“入口合并、能力不合并”**：在 UI 上做一个统一导入入口，但保留两条处理链路（Excel 直导链路 + 文本/Word 解析链路）。
- 原因：用户体验统一，同时避免把稳定的 Excel 导入能力绑死在高波动的解析逻辑上。

## 目标
- 回答“是否有必要合并”并给出可执行落地方案。
- 在不牺牲稳定性的前提下，降低教师使用复杂度与维护成本。

## 合并对象拆解
- 现有能力A：**批量导入题目（Excel）**
  - 优势：结构化、可审计、失败定位精确、技术风险低。
- 现有能力B：**一键解析并导入（docx/txt/粘贴）**
  - 优势：输入门槛低、贴合教师真实材料形态。
  - 风险：格式多样、解析误差、异常处理复杂。

## 方案比较

### 方案1：完全合并为单一后端接口（不推荐）
- 做法：所有输入最终都走同一个接口和同一处理器。
- 风险：
  - 解析异常会影响 Excel 稳定链路；
  - 回归测试面扩大，故障定位困难；
  - 发布风险高。

### 方案2：入口合并 + 链路分治（推荐）
- 做法：
  - 前端仅保留一个“导入题库”入口；
  - 在入口内按输入类型切换：
    - Excel -> 走现有批量导入接口；
    - docx/txt/粘贴 -> 走一键解析接口；
  - 展示统一的导入结果面板（成功数/失败数/错误明细）。
- 价值：
  - 用户心智统一（一个入口）；
  - 系统稳定性保留（两条链路隔离）；
  - 后续可灰度优化解析器，不影响 Excel。

### 方案3：保持两个独立入口（可行但次优）
- 做法：维持“批量导入”与“一键解析并导入”两个按钮。
- 问题：用户选择成本高，容易误用，产品层面不够简洁。

## 推荐落地步骤（按方案2）

### 第1步：统一交互入口
- 将页面按钮收敛为“导入题库”主入口。
- 入口内提供“输入方式”切换：Excel / Word或文本。
- 默认记住上次选择，减少重复操作。

### 第2步：统一结果模型
- 规范返回结构：
  - `message`
  - `successCount`
  - `failedCount`
  - `errors[]`
- 前端统一渲染结果，避免两套提示逻辑。

### 第3步：统一校验与兜底
- 前置校验统一：
  - 课程必选；
  - 文件类型合法；
  - 文本非空。
- 解析链路失败时，给“转 Excel 模板导入”兜底动作。

### 第4步：链路隔离与可观测性
- 保持两个后端端点不互相依赖。
- 增加导入日志字段：入口类型、文件类型、解析失败原因分类、耗时、成功率。
- 便于后续判断是否要进一步深度合并。

### 第5步：验收标准
- 用户只感知一个导入入口。
- Excel 导入成功率与性能不低于现状。
- 文本/Word 导入失败有明确原因和可执行下一步。
- 两类导入结果展示一致，支持一键重试失败项（后续增强项）。

## 是否“有必要合并”的最终判断
- **有必要在产品层面合并（入口统一）**；
- **不建议在技术实现层面立即硬合并（处理链路保留分离）**。
- 这是当前阶段在“体验、稳定、成本”三者之间的最优解。
